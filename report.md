# Отчёт По Проекту MiniC

## 1. Общее описание

В рамках проекта реализован небольшой учебный компилятор для языка MiniC на
C++23. Компилятор построен как фазовый pipeline:

```text
source -> lexer -> parser -> semantic analyzer -> code generator
```

Цель проекта состояла в том, чтобы построить полный hand-written compiler
pipeline для собственного языка с собственной спецификацией, парсером,
семантическим анализатором и x86-64 backend.

## 2. Архитектура

### Lexer

Лексер преобразует исходный текст в поток токенов с позициями в файле. Он
распознаёт ключевые слова, идентификаторы, литералы, операторы, разделители и
комментарии. В текущей версии дополнительно поддерживаются вложенные блочные
комментарии `/* ... */`.

### Parser

Парсер реализован как recursive descent parser. Он потребляет поток токенов и
строит AST, описывающее объявления, инструкции, выражения и синтаксис типов.
Конвейерный оператор `|>` десугарится уже на этапе парсинга в обычный вызов
функции.

### Semantic analysis

Семантический анализатор разрешает имена, проверяет области видимости,
валидирует типы, проверяет правила control-flow и формирует semantic result,
эквивалентный annotated AST. На этом же этапе проверяются побитовые операции и
сдвиги над целочисленными типами.

### Code generation

Backend генерирует x86-64 assembly в Intel syntax и при наличии подходящего
toolchain умеет собирать готовый `x86-64` executable. Он поддерживает
scalar-код, локальные массивы, локальные структуры, aggregate copy semantics,
builtins и небольшой набор runtime helpers, вынесенных во внешний runtime-файл.
Дополнительно backend умеет генерировать код для побитовых операций и сдвигов.

## 3. Основные проектные решения

- Recursive descent был выбран потому, что он естественно отображается на
  грамматику проекта.
- Фаза семантики отделена от парсинга и генерации кода.
- Был выбран прямой backend `AST -> assembly`, без введения отдельного IR.
- Runtime helpers вынесены из `codegen.cpp` во внешний файл
  `runtime/minic_runtime.s`, чтобы отделить генерацию пользовательского кода от
  служебной runtime-логики.
- Язык использует value semantics для всех присваиваний и передачи аргументов.

## 4. Реализованные возможности

Текущая реализация включает:

- собственную спецификацию языка
- hand-written lexer
- recursive descent parser
- semantic analyzer
- генерацию x86-64 assembly
- builtins:
  - `print`
  - `input`
  - `len`
  - `exit`
  - `panic`
  - `assert`
- floating-point code generation
- `char`, символьные литералы и `cast<string>(char)`
- `0x` / `0b` integer literals, exponent float literals, `inf`, `NaN`
- aggregate-параметры функций и aggregate-return
- equality для массивов и структур
- `cast<string>(numeric-or-bool-or-char)`

Дополнительно как отдельные задания реализованы:

- `A.1.12` вложенные блочные комментарии `/* ... */`
- `B.1.2` расширенные диагностические сообщения с показом строки и подчёркиванием `^`
- `B.1.5` дополнительные CLI-режимы `--emit-asm` и `--time-phases`
- `A.1.8` побитовые операции `& | ^ ~ << >>`
- `A.1.9` конвейерный оператор `|>`
- `A.2.1` выражение `if`
- `A.2.3` методы структур
- `A.2.4` видимость членов структуры `public/private`
- `A.2.9` перегрузка функций
- `A.2.10` параметры по умолчанию и именованные аргументы
- `A.2.11` перегрузка операторов
- `A.2.13` модули
- `A.3.14` полноценная модульная система с контролем видимости

### Подробное описание дополнительных заданий

- `A.1.12` вложенные блочные комментарии:
  лексер поддерживает комментарии вида `/* ... */` с произвольной глубиной
  вложенности. Для этого во время пропуска комментария поддерживается счётчик
  глубины, который увеличивается на новом `/*` и уменьшается на `*/`.
- `B.1.2` расширенные диагностические сообщения:
  все основные фазы front-end возвращают ошибки с координатами в исходнике, а
  форматирование диагностик дополняется выводом строки с ошибкой и
  подчёркиванием позиции символом `^`. Это делает сообщения lexer/parser/semantic
  заметно удобнее для отладки.
- `B.1.5` дополнительные CLI-режимы:
  флаг `--emit-asm` позволяет остановиться после codegen и сохранить только
  сгенерированный `x86-64` assembly, а `--time-phases` выводит время отдельных
  фаз pipeline. Эти режимы полезны и для отладки, и для тестирования.
- `A.1.8` побитовые операции:
  в язык добавлены `& | ^ ~ << >>`. Они поддержаны в parser, проверяются в
  semantic phase как операции только над целочисленными типами и напрямую
  генерируются backend'ом.
- `A.1.9` конвейерный оператор:
  выражение `value |> f |> g(x)` разбирается parser'ом и десугарится в обычные
  вложенные вызовы функций. За счёт этого в семантике и codegen такой оператор
  ведёт себя как обычный call expression.
- `A.2.1` выражение `if`:
  в AST добавлен отдельный узел для `if`-expression. Семантика требует тип
  `bool` у условия и совпадение типов обеих ветвей, а codegen генерирует
  ветвление с вычислением итогового значения выражения.
- `A.2.3` методы структур:
  `struct` может содержать методы, которые регистрируются как специальные
  `FunctionSymbol` с флагом `is_method`. На этапе semantic-call-binding к ним
  автоматически подставляется скрытый `self`, а backend передаёт `self` как
  первый аргумент вызова.
- `A.2.4` `public/private`:
  для полей и методов структур реализованы секции видимости. Семантика хранит
  visibility в `FieldSymbol` и `FunctionSymbol`, запрещает доступ к private
  членам извне структуры и выдаёт отдельные диагностические сообщения для таких
  случаев.
- `A.2.9` перегрузка функций:
  функции хранятся не по одному имени, а как наборы overload-кандидатов. При
  вызове семантика перебирает все `FunctionSymbol` с данным именем, проверяет
  их сигнатуры и выбирает единственный подходящий вариант либо сообщает об
  ошибке `no match`/`ambiguous`.
- `A.2.10` параметры по умолчанию и именованные аргументы:
  вызов функции теперь может смешивать positional и named arguments по правилам
  языка, а недостающие параметры могут заполняться `default value`. Эта логика
  сосредоточена в `bind_call_arguments(...)`, который строит итоговый список
  аргументов для semantic-check и codegen.
- `A.2.11` перегрузка операторов:
  для unary/binary операторов реализован поиск пользовательских operator-method
  и свободных `operator...`-функций. После обычных builtin-правил семантика
  выполняет overload resolution, запоминает выбранную перегрузку в
  `SemanticResult`, а codegen затем просто генерирует обычный вызов уже
  разрешённой функции.
- `A.2.13` модули:
  поддержаны объявления `module` и `import`. Front-end умеет находить файл
  модуля по имени, рекурсивно загружать импортированные зависимости и встраивать
  импортированные декларации в общее дерево программы.
- `A.3.14` полноценная модульная система с контролем видимости:
  поверх базовых модулей добавлены `export`, `private` и выборочный импорт
  `import Module::name`. При загрузке модуля импортируются только экспортируемые
  декларации, а попытка импортировать или вызвать скрытое объявление приводит к
  отдельной семантической ошибке.

## 5. Известные ограничения

Главное оставшееся ограничение находится уже не в core language pipeline, а в
окружении сборки:

- финальная генерация готового executable зависит от host toolchain и менее
  переносима, чем генерация assembly на машинах без `x86-64` linker/assembler

## 6. Сборка и запуск

Сборка:

```bash
cmake -S . -B build
cmake --build build
```

Запуск:

```bash
./build/minic examples/semantic_ok.mc
./build/minic examples/semantic_ok.mc --dump-tokens
./build/minic examples/semantic_ok.mc --dump-ast
./build/minic examples/pipe_ok.mc --emit-asm --time-phases -o examples/pipe_ok.s
```

## 7. Подход к тестированию

Проект проверялся на следующих наборах примеров:

- корректные full-pipeline примеры
- примеры с лексическими ошибками
- примеры с синтаксическими ошибками
- примеры с семантическими ошибками
- примеры на aggregate code generation
- примеры на floating-point code generation
- примеры на вложенные блочные комментарии
- примеры на побитовые операции и сдвиги
- примеры на конвейерный оператор
- примеры на выражение `if`
- примеры на методы структур и видимость
- примеры на перегрузку функций, named args и default args
- примеры на перегрузку операторов
- примеры на модульный импорт
- примеры на `export` / `private` в модулях и выборочный импорт
- проверка расширенных диагностик на ошибочных входных файлах

Дополнительно в каталоге `tests/` собран отдельный набор smoke/integration
примеров для всех реализованных допов. Там находятся:

- положительные программы для nested comments, bitwise ops, pipe operator и
  `if`-expression
- положительные программы для методов структур, `public/private`, overloads,
  named/default args и operator overloading
- положительные программы для модулей и выборочного импорта
- отрицательные программы на доступ к private-методу и private-объявлениям
  модулей
- скрипт `tests/run_tests.sh`, который автоматически прогоняет эти кейсы в
  compile-to-asm режиме, а при поддерживаемом host toolchain может использоваться
  и для полного end-to-end запуска

  Отдельно в `examples/graph_traversal_bfs_dfs.mc` добавлена бонусная программа
из пула заданий: обход фиксированного графа и в ширину (BFS), и в глубину
(DFS). Она демонстрирует работу массивов, индексации, циклов, функций и
печати результатов на пользовательской программе, написанной уже на самом
языке MiniC.

## 8. Итог

Проект уже реализует полноценный compiler pipeline и генерирует x86-64 assembly
для значимой части языка. Следующим естественным шагом развития может быть
улучшение переносимой сборки готового executable и дальнейшее упрощение
deployment-пути для разных host environments.
