# Генератор кода MiniC

Генератор кода MiniC реализован как последняя фаза pipeline:

```text
Lexer -> Parser -> Semantic -> Codegen
```

На вход codegen получает AST и результат семантического анализа
(`SemanticResult`), содержащий аннотации типов и сведения о разрешённых
функциях и структурах.

---

## 1. Целевая платформа и формат вывода

### 1.1 Выбранная платформа

Генератор кода ориентирован на:

- архитектуру `x86-64`
- соглашение о вызовах System V AMD64 ABI
- текстовый output в виде assembly-файла

### 1.2 Формат вывода

Результатом codegen является текст assembly в Intel-синтаксисе для GNU assembler:

- директива `.intel_syntax noprefix`
- секции `.text` и `.rodata`
- имена внешних функций через `.extern`
- вызовы внешнего runtime из файла `runtime/minic_runtime.s`

Если в окружении доступен совместимый `x86-64` assembler/linker,
компилятор по умолчанию пытается собрать готовый executable.
Если toolchain недоступен или явно передан `--emit-asm`,
результат сохраняется в файл с расширением `.s`.

Пример:

```text
minic hello.mc --emit-asm
```

создаёт:

```text
hello.s
```

Также поддерживается:

```text
minic hello.mc -o out.s
```

Дополнительно для управления стадией backend поддерживаются флаги:

- `--emit-asm` — остановиться на выводе assembly;
- `--time-phases` — напечатать время фаз `lex`, `parse`, `semantic`, `codegen`, `link`.

---

## 2. Архитектура backend

### 2.1 Общая схема

В текущей реализации используется прямой backend:

```text
AST + SemanticResult -> x86-64 assembly
```

Промежуточное представление (IR) не вводится.
Оптимизации в отдельную фазу не выделяются.

### 2.2 Почему выбран прямой backend

Этот подход:

- проще для учебного проекта;
- позволяет быстро получить рабочую генерацию без отдельного IR;
- хорошо подходит для небольшого языка со строгой грамматикой и value semantics.

---

## 3. Входные данные для codegen

Из `SemanticResult` генератор использует:

- тип каждого выражения;
- сигнатуры функций;
- полные имена функций;
- сведения о структурах и их полях;
- типы локальных переменных.

Это позволяет:

- корректно выбирать инструкции сравнения для signed/unsigned типов;
- вычислять размер массивов и структур;
- генерировать копирование по значению;
- различать builtin-функции и обычные пользовательские функции;
- получать уже десугаренные вызовы для конвейерного оператора `|>`.

---

## 4. Соглашение о вызовах

### 4.1 Передача аргументов

Первые шесть аргументов функции передаются через регистры:

- `rdi`
- `rsi`
- `rdx`
- `rcx`
- `r8`
- `r9`

Текущая реализация поддерживает не более 6 параметров и не более 6 аргументов вызова.

### 4.2 Возвращаемое значение

- скалярный результат функции возвращается в `rax`
- `void`-функция не возвращает значимого значения

### 4.3 Поддерживаемые типы параметров и возврата

Сейчас кодогенератор полностью поддерживает скалярные типы:

- `int8`, `int16`, `int32`, `int64`
- `uint8`, `uint16`, `uint32`, `uint64`
- `bool`
- `char`
- `string`
- `float32`, `float64`

Также поддерживаются:

- aggregate-параметры функций (`struct`, `array`)
- aggregate-return через скрытый destination pointer

---

## 5. Организация стека и локальных данных

### 5.1 Stack frame

Для каждой функции строится обычный stack frame:

```text
push rbp
mov rbp, rsp
sub rsp, <stack_size>
```

Локальные переменные и временные слоты располагаются в стеке по адресам вида:

```text
[rbp - offset]
```

### 5.2 Локальные переменные

- скалярные локальные переменные занимают 8 байт;
- временные значения для вычисления выражений тоже выделяются в стеке;
- массивы и структуры размещаются как непрерывные области памяти внутри stack frame.

### 5.3 Выравнивание

Общий размер stack frame выравнивается по 16 байтам перед вызовами функций.

---

## 6. Представление типов в памяти

### 6.1 Скалярные типы

В runtime backend нормализует скалярные значения к 64-битному хранению:

- `int8`, `int16`, `int32` знаково расширяются;
- `uint8`, `uint16`, `uint32` беззнаково расширяются;
- `int64`, `uint64`, `bool`, `char`, `string` уже укладываются в 64-битную модель хранения.

Строка представляется как указатель на null-terminated последовательность байт.

### 6.2 Массивы

Массив хранится как непрерывный блок памяти.

- элементы расположены подряд;
- размер массива известен на этапе компиляции;
- индексирование вычисляет смещение как `index * sizeof(element)`.

### 6.3 Структуры

Поля структуры хранятся в порядке объявления.

- доступ к полю реализуется как смещение от базового адреса структуры;
- присваивание структуры копирует весь непрерывный блок памяти структуры.

---

## 7. Генерация инструкций

### 7.1 Функции

Для каждой пользовательской функции генерируется отдельная метка.

Полные имена функций mangled в вид:

```text
__minic_fn_<name>
```

Для `main` имя сохраняется как `main`.

Квалифицированные имена из `namespace` тоже mangle-ятся в допустимый label.

### 7.2 Арифметика

Для целочисленной арифметики используются:

- `add`
- `sub`
- `imul`
- `idiv` / `div`

Перед делением генерируется runtime-проверка деления на ноль.

### 7.3 Побитовые операции и сдвиги

Для целочисленных побитовых операций используются:

- `and`
- `or`
- `xor`
- `not`

Для `<<` и `>>` используются:

- `shl` для левого сдвига;
- `shr` для беззнакового правого сдвига;
- `sar` для знакового правого сдвига.

Если величина сдвига отрицательна или не меньше разрядности типа,
codegen генерирует результат `0`, а не пытается выполнять платформенно-зависимый сдвиг.

### 7.4 Логика

Логические операции `&&` и `||` генерируются с short-circuit semantics через labels и jumps.

Оператор `!` генерирует сравнение с нулём и установку результата через `sete`.

### 7.5 Сравнения

Сравнения генерируются через:

- `cmp`
- `sete`, `setne`
- `setl`, `setle`, `setg`, `setge`
- `setb`, `setbe`, `seta`, `setae` для unsigned-сравнений

Для строкового `==` / `!=` используется вызов `strcmp`.
Для массивов и структур equality генерируется рекурсивно по элементам и полям.

### 7.6 Условные и циклы

- `if` / `else` генерируются через labels и условные переходы;
- `while` генерируется как цикл с отдельными метками начала, тела и выхода;
- `break` прыгает на label выхода из текущего цикла;
- `continue` прыгает на label повторной проверки условия.

### 7.7 Присваивание и value semantics

Скалярное присваивание:

- вычисляет адрес l-value;
- вычисляет правую часть;
- записывает значение по адресу.

Для массивов и структур используется value semantics:

- литералы и локальные агрегаты записываются в выделенную память;
- копирование aggregate-значений выполняется через `memcpy`.

### 7.7 Доступ к элементу массива

Для `a[i]`:

- вычисляется адрес массива;
- вычисляется индекс;
- выполняется runtime-проверка выхода за границы;
- затем вычисляется адрес элемента.

### 7.8 Доступ к полю структуры

Для `s.field`:

- вычисляется адрес базовой структуры;
- к нему прибавляется compile-time смещение поля.

---

## 8. Литералы и секция `.rodata`

Строковые литералы интернируются и помещаются в `.rodata` как:

```asm
.asciz "..."
```

Повторяющиеся строковые литералы переиспользуют уже созданную метку.

---

## 9. Builtin-функции и runtime helpers

### 9.1 Внешние зависимости

Сгенерированный assembly использует внешние runtime-функции платформы:

- `printf`
- `strlen`
- `strcmp`
- `malloc`
- `memcpy`
- `exit`

### 9.2 Встроенные функции языка

Codegen обрабатывает встроенные функции отдельно:

- `print`
- `input`
- `len`
- `exit`
- `panic`
- `assert`

### 9.3 Внешний runtime

Служебные runtime-функции вынесены из `codegen.cpp` во внешний файл:

```text
runtime/minic_runtime.s
```

Кодоген не генерирует их тела внутри выходного файла программы, а только
ссылается на них через вызовы `__minic_rt_*`.

Во внешнем runtime реализованы:

- `__minic_rt_print_i64`
- `__minic_rt_print_u64`
- `__minic_rt_print_string`
- `__minic_rt_print_bool`
- `__minic_rt_print_char`
- `__minic_rt_print_f32`
- `__minic_rt_print_f64`
- `__minic_rt_bool_to_string`
- `__minic_rt_char_to_string`
- `__minic_rt_i64_to_string`
- `__minic_rt_u64_to_string`
- `__minic_rt_f32_to_string`
- `__minic_rt_f64_to_string`
- `__minic_rt_input`
- `__minic_rt_panic`
- `__minic_rt_div_zero`
- `__minic_rt_index_out_of_bounds`
- `__minic_rt_concat`

Эти функции отвечают за:

- печать целых значений;
- печать unsigned-значений;
- печать строк;
- печать `bool`;
- печать `char`;
- печать floating-point значений;
- приведение scalar-значений к `string`;
- чтение строки из `stdin`;
- вывод panic-message и завершение программы;
- сообщение о делении на ноль;
- сообщение о выходе за границы массива;
- конкатенацию строк.

---

## 10. Поддерживаемые конструкции

Текущая реализация codegen поддерживает:

- функции с не более чем 6 register arguments
  (с учётом скрытого aggregate-return pointer, если он нужен);
- локальные переменные;
- `if`, `while`, `break`, `continue`, `return`;
- целочисленные арифметические операции;
- побитовые операции и сдвиги над целочисленными типами;
- floating-point арифметические операции;
- логические операции и сравнения;
- строковые литералы;
- конкатенацию строк;
- конвейерный оператор `|>` после десугаринга в обычный вызов;
- локальные `struct` и массивы фиксированного размера;
- `struct` literals;
- `array` literals;
- доступ `.` и `[]`;
- value-copy массивов и структур;
- aggregate-параметры функций;
- aggregate-return через скрытый destination pointer;
- explicit numeric casts;
- `cast<string>(numeric-or-bool-or-char)`;
- вызовы `print`, `input`, `len`, `exit`, `panic`, `assert`.

---

## 11. Ограничения текущей реализации

На текущем этапе главным ограничением остаётся полноценная стадия assemble/link:

- компилятор всегда умеет генерировать `.s`;
- генерация готового executable зависит от host toolchain и доступности
  совместимого x86-64 assembler/linker.

---

## 12. Возможные дальнейшие улучшения

В следующих версиях backend можно добавить:

- более переносимую стадию линковки исполняемого файла;
- простые оптимизации;
- переход на собственное IR перед генерацией assembly.
