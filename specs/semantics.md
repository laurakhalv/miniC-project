# Семантика языка MiniC

Семантический анализ выполняется после построения AST и проверяет корректность
программы на уровне смысла: области видимости, типы, вызовы функций,
управляющие конструкции и корректность встроенных операций.

Семантический анализатор реализуется как отдельная фаза pipeline и формирует
аннотированный AST или эквивалентный ему результат анализа, который затем
используется генератором кода.

---

## 1. Модель выполнения

- Каждый исходный файл языка MiniC является отдельным модулем.
- Одна компиляция обрабатывает один модуль.
- Точка входа программы - функция `main` с сигнатурой `func main() -> int32`.
- Возвращаемое значение `main` используется как код завершения процесса.
- Аргументы функций передаются по значению.
- Функции могут иметь побочные эффекты.

---

## 2. Области видимости и разрешение имён

Язык использует лексическую область видимости.

### 2.1 Где что можно объявлять

- На верхнем уровне модуля и внутри `namespace` допускаются только объявления:
  функции, структуры, синонимы типов и вложенные пространства имён.
- Внутри `struct` допускаются поля, методы и секции видимости `public:` / `private:`.
- Внутри тела функции допускаются инструкции и локальные объявления переменных.
- Глобальные переменные и переменные на уровне `namespace` не поддерживаются.

### 2.2 Правила разрешения имён

- Идентификатор разрешается в ближайшей охватывающей области видимости.
- Использование имени до его объявления запрещено.
- Два одинаковых имени в одной и той же области видимости запрещены, кроме допустимой перегрузки функций.
- Затенение имён во вложенных блоках разрешено.
- Пространства имён участвуют в разрешении имён только для объявлений верхнего уровня.
- Внутри метода неявно доступна переменная `self` типа текущей структуры.
- Имя перегруженной функции нельзя использовать как самостоятельное значение без вызова, если по нему существует больше одной сигнатуры.
- Импортированный модуль делает все свои верхнеуровневые объявления доступными через квалифицированное имя модуля.

Пример:

```minic
func main() -> int32 {
    let int32 x = 0;

    {
        let int32 x = 1;
        print(cast<string>(x));
    }

    let int32 y = x;
    return y;
}
```

### 2.3 Модули

- Файл может объявить себя модулем через `module Name;`.
- Файл может импортировать другой модуль через `import Name;`.
- Файл может импортировать отдельное экспортированное объявление через `import Name::foo;`.
- Верхнеуровневое объявление модуля может быть помечено как `export` или `private`.
- После импорта модуля целиком доступны только экспортированные top-level объявления.
- После импорта модуля целиком объявления модуля доступны через квалифицированные имена вида `Name::foo`.
- После выборочного импорта экспортированное объявление становится доступным напрямую по своему имени.
- В текущей реализации импорт модуля десугарится в синтетическое пространство имён с тем же именем.

## 3. Блоки, переменные и время жизни

### 3.1 Блок

- Блок `{ ... }` создаёт новую локальную область видимости.
- Все локальные переменные живут только до конца блока, в котором объявлены.

### 3.2 Пустая инструкция

- Инструкция `;` не производит никакого эффекта.

### 3.3 Переменные

- `let` объявляет мутабельную переменную.
- `const` объявляет иммутабельную переменную.
- Инициализация обязательна.
- Использование неинициализированной переменной невозможно, так как объявление без инициализатора запрещено.
- Переменная типа `void` запрещена.
- Повторное присваивание `const`-переменной является семантической ошибкой.

### 3.4 Модель памяти

- Скалярные значения хранятся как самостоятельные значения и копируются целиком.
- Строка семантически ведёт себя как immutable string value, а backend хранит её как указатель
  на null-terminated данные.
- Массивы и структуры используют value semantics:
  - присваивание копирует весь массив или всю структуру;
  - передача аргумента функции копирует всё значение;
  - возврат aggregate-значения копирует результат в выделенную область вызывающей стороны.
- Элементы массива хранятся подряд, а поля структуры располагаются в порядке объявления.

## 4. Система типов

### 4.1 Общие свойства

- Язык статически типизирован.
- Типизация номинальная.
- Неявные приведения запрещены.
- Все значения передаются и присваиваются по значению.
- Синоним типа (`type`) не создаёт новый тип.

### 4.2 Встроенные типы

Поддерживаются:

- `int8`, `int16`, `int32`, `int64`
- `uint8`, `uint16`, `uint32`, `uint64`
- `float32`, `float64`
- `bool`
- `char`
- `string`
- `void`

### 4.3 Составные типы

- `T[N]` - массив фиксированного размера.
  Размер входит в тип, поэтому `int32[3]` и `int32[4]` являются разными типами.
- `struct` - пользовательский составной тип.
  Две структуры с одинаковыми полями, но разными именами, считаются разными типами.

### 4.4 Тип `void`

- `void` допустим только как тип возврата функции.
- Переменные, параметры, элементы массива и поля структуры типа `void` запрещены.

### 4.5 Методы и видимость

- Метод семантически является функцией, привязанной к структуре.
- Метод получает неявный первый параметр `self` типа текущей структуры.
- Доступ к `private`-полям и `private`-методам разрешён только из методов той же структуры.
- Снаружи структуры разрешён доступ только к `public`-членам.

## 5. Числовые литералы и переполнение

### 5.1 Типизация литералов

- `BOOL_LITERAL` имеет тип `bool`.
- `CHAR_LITERAL` имеет тип `char`.
- `STRING_LITERAL` имеет тип `string`.
- `INT_LITERAL` семантически трактуется как целочисленный литерал.
  Если контекст ожидает конкретный целочисленный тип, литерал получает этот тип,
  если значение представимо в нём. Если контекст не задаёт тип, используется `int32`.
  Поддерживаются десятичные, шестнадцатеричные (`0x...`) и двоичные (`0b...`) формы.
- `FLOAT_LITERAL` семантически трактуется как вещественный литерал.
  Если контекст ожидает `float32` или `float64`, литерал получает ожидаемый тип,
  если значение представимо в нём. Если контекст не задаёт тип, используется `float64`.
  Поддерживаются экспоненциальная запись и специальные значения `inf`, `-inf`, `NaN`.

Это правило относится к типизации литералов и не считается неявным приведением.

### 5.2 Overflow

- Целочисленное переполнение не проверяется во время выполнения.
- Результат целочисленной арифметики вычисляется по модулю `2^N`, где `N` - размер типа.
- Поведение вещественных операций при overflow и underflow следует поведению целевой платформы IEEE 754.

---

## 6. Явные приведения типов

Допускается только явное приведение:

```minic
cast<TargetType>(expression)
```

Разрешены следующие преобразования:

- любой числовой тип -> любой числовой тип
- любой числовой тип -> `string`
- `bool` -> `string`
- `char` -> `string`

Запрещены, например:

- `string` -> `int32`
- `struct` -> `int32`
- `array` -> `string`

Приведение несовместимых типов является семантической ошибкой.

## 7. Семантика выражений

### 7.1 Арифметика

- Для `-`, `*`, `/`, `%` оба операнда должны быть одного и того же числового типа.
- Результат имеет тот же тип.
- `+` поддерживает:
  - сложение двух значений одного и того же числового типа;
  - конкатенацию двух строк.
- Деление на ноль является ошибкой времени выполнения.

### 7.2 Логика

- `&&`, `||` и `!` применимы только к `bool`.
- Результат имеет тип `bool`.

### 7.3 Побитовые операции

- `&`, `|`, `^` применимы только к целочисленным типам.
- Оба операнда должны иметь один и тот же целочисленный тип.
- Результат имеет тот же тип.
- Унарный `~` применим только к целочисленному типу и возвращает значение того же типа.

### 7.4 Сдвиги

- `<<` и `>>` применимы только к целочисленным типам.
- Оба операнда должны иметь один и тот же целочисленный тип.
- Результат имеет тот же тип.
- Если величина сдвига отрицательна или не меньше разрядности типа, результатом считается `0`.
- Для `>>` над беззнаковыми типами используется логический сдвиг, над знаковыми — арифметический.

### 7.5 Сравнения

- `==` и `!=` разрешены для:
  - числовых типов
  - `bool`
  - `char`
  - `string`
  - массивов
  - структур
- `<`, `>`, `<=`, `>=` разрешены только для числовых типов.
- Операнды должны иметь одинаковые допустимые типы.
- Результат сравнений всегда `bool`.

### 7.6 Конвейерный оператор

- `x |> f` является синтаксическим сахаром и семантически эквивалентен `f(x)`.
- `x |> f(a, b)` семантически эквивалентен `f(x, a, b)`.
- После десугаринга к полученному вызову применяются обычные правила проверки аргументов и типа возврата.

### 7.7 Вызов функции

- Вызов разрешается по имени функции и списку типов аргументов.
- Для одного имени может существовать несколько перегруженных сигнатур.
- Если подходит ровно одна сигнатура, вызывается она.
- Если не подходит ни одна сигнатура, это семантическая ошибка.
- Если подходят несколько сигнатур, вызов считается неоднозначным и это семантическая ошибка.
- Именованные аргументы сопоставляются по имени параметра.
- После первого именованного аргумента позиционные аргументы запрещены.
- Один и тот же параметр нельзя передать дважды.
- Если аргумент для параметра отсутствует, используется значение по умолчанию, если оно объявлено.
- После параметра со значением по умолчанию не может идти параметр без значения по умолчанию.
- После связывания аргументов количество фактических аргументов должно совпадать с количеством параметров.
- Тип каждого фактического аргумента должен совпадать с типом параметра.
- Аргументы вычисляются слева направо.
- Тип выражения вызова равен типу возврата вызываемой функции.

Вызов метода `obj.method(x)` семантически связывается как обычный вызов метода структуры с неявной передачей `obj` в `self`.

### 7.8 Присваивание

- Левая часть должна быть L-value:
  - идентификатор переменной
  - доступ к элементу массива
  - доступ к полю структуры
- Простое `NamespaceAccess` не является L-value.
- Тип правой части должен совпадать с типом левой части.
- Присваивание `const`-переменной запрещено.
- Присваивание использует value semantics:
  - при присваивании массива копируется весь массив;
- при присваивании структуры копируется вся структура;
- при присваивании скалярного значения копируется само значение.

### 7.9 Индексирование массива

- Базовое выражение должно иметь тип массива.
- Индекс должен иметь один из целочисленных типов.
- Результат имеет тип элемента массива.
- Выход за границы массива проверяется во время выполнения.

### 7.10 Доступ к полю структуры

- Базовое выражение должно иметь тип структуры.
- Поле должно существовать в этой структуре.
- Результат имеет тип данного поля.
- Обращение к `private`-полю вне методов этой же структуры является семантической ошибкой.

### 7.11 Доступ через пространство имён

- `A::B::name` разрешается как обращение к объявлению внутри пространства имён.
- Такое выражение является r-value.
- Оно может использоваться как callee перед вызовом, например `Math::square(2)`.
- Оно не может быть целью присваивания.

### 7.12 Литерал массива

- Все элементы должны иметь один и тот же тип.
- Если литерал используется в контексте типа `T[N]`, количество элементов должно быть равно `N`.
- Тип каждого элемента должен совпадать с типом `T`.
- Если литерал массива используется без контекстно заданного типа, его тип выводится как `T[N]`,
  где `T` - тип первого элемента, а `N` - число элементов.
- Пустой литерал массива без контекстно заданного типа запрещён.

### 7.13 Выражение `if`

- Условие в `if`-выражении должно иметь тип `bool`.
- Типы ветвей `then` и `else` должны совпадать.
- Тип всего `if`-выражения равен типу обеих ветвей.
- В отличие от инструкции `if`, у выражения `if` ветвь `else` обязательна.

### 7.14 Перегрузка операторов

- Пользовательский тип может объявлять методы-операторы или свободные функции-операторы.
- Перегруженный оператор разрешается по имени оператора и списку типов аргументов.
- Если встроенное правило для оператора применимо, используется оно.
- Если встроенное правило не применимо, семантика пытается найти пользовательскую перегрузку.
- Для бинарного оператора метод структуры получает левый операнд как `self`, а правый — как обычный параметр.
- Для унарного оператора метод структуры получает единственный неявный параметр `self`.
- Если подходит ровно одна перегрузка, используется она.
- Если не подходит ни одна, это семантическая ошибка.
- Если подходят несколько, оператор считается неоднозначным.
- Для `private`-методов-операторов действуют обычные правила видимости.

### 7.13 Литерал структуры

- Тип литерала задаётся именем структуры перед `{ ... }`.
- Допустимы и квалифицированные имена вида `Math::Point`.
- Все поля структуры должны быть инициализированы ровно один раз.
- Лишние поля запрещены.
- Тип каждого значения должен совпадать с типом соответствующего поля.

### 7.11 Порядок вычисления

- Подвыражения вычисляются слева направо.
- В бинарных выражениях сначала вычисляется левый операнд, затем правый.
- В вызове функции сначала вычисляется callee, затем аргументы слева направо.

## 8. Управляющие конструкции

### 8.1 `if` / `else`

- Условие должно иметь тип `bool`.
- Ветка `else` необязательна.

### 8.2 `while`

- Условие должно иметь тип `bool`.

### 8.3 `break` и `continue`

- Допустимы только внутри тела цикла `while`.
- Использование вне цикла является семантической ошибкой.

### 8.4 `return`

- В функции с типом возврата, отличным от `void`:
  - `return` обязан содержать выражение;
  - тип выражения должен совпадать с типом возврата;
  - все пути выполнения должны завершаться `return`.
- В функции с типом возврата `void`:
  - разрешён `return;`
  - отсутствие явного `return` в конце тела допустимо.
- `return` с выражением в `void`-функции запрещён.

---

## 9. Функции

- Перегрузка функций не поддерживается.
- Рекурсия разрешена.
- Параметры функции ведут себя как локальные иммутабельные значения, переданные по значению.
- Имена параметров внутри одной функции должны быть уникальны.
- Имя функции не должно конфликтовать с другим объявлением в той же области видимости.

### 9.1 `main`

- Программа обязана содержать `main`.
- `main` не принимает параметров.
- `main` обязан возвращать `int32`.

---

## 10. Встроенные функции

Встроенные функции доступны без объявления и разрешаются семантическим анализатором.

| Сигнатура | Описание |
|-----------|----------|
| `print(T) -> void` | вывод значения в stdout |
| `input() -> string` | чтение строки из stdin |
| `len(string) -> int32` | длина строки |
| `len(T[N]) -> int32` | возвращает известную на этапе компиляции длину массива |
| `exit(int32) -> void` | немедленное завершение программы |
| `panic(string) -> void` | аварийное завершение программы |
| `assert(bool) -> void` | аварийное завершение при ложном условии |

Для `print(T)` допустимы:

- любой целочисленный тип
- любой вещественный тип
- `bool`
- `char`
- `string`

`exit` и `panic` завершают выполнение программы и не возвращают управление.
`assert(false)` завершается через тот же runtime panic-механизм.

---

## 11. Ошибки времени выполнения

Во время выполнения обязательно обрабатываются:

- деление на ноль
- выход за границы массива

Формат сообщения:

```text
runtime error: <сообщение> at line <N>
```

Примеры:

- `runtime error: division by zero at line 12`
- `runtime error: index out of bounds at line 7`

---

## 12. Семантические ошибки

Сообщения о compile-time ошибках выводятся в расширенном формате:

```text
<файл>:<строка>:<столбец>: error: <сообщение>
<строка исходного кода>
<пробелы>^
```

Примеры семантических ошибок:

- использование необъявленного имени
- повторное объявление имени в одной области видимости
- несовпадение типов
- присваивание в `const`
- `break` или `continue` вне цикла
- отсутствие `main`
- неверная сигнатура `main`
- отсутствие `return` на одном из путей выполнения
- обращение к несуществующему полю
- обращение к несуществующему имени в `namespace`
- приведение несовместимых типов

## 13. Порядок семантического анализа

Семантический анализ выполняется в два прохода.

### Проход 1

Собираются все глобальные объявления модуля:

- функции
- структуры
- псевдонимы типов
- пространства имён

Это позволяет функциям ссылаться друг на друга независимо от порядка объявления.

### Проход 2

Проверяются тела функций:

- локальные области видимости
- типы выражений
- вызовы функций
- `return`
- `break` / `continue`
- литералы массивов и структур
- встроенные функции
